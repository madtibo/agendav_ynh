# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is a **YunoHost packaging wrapper** for AgenDAV (a CalDAV web client), **not** the AgenDAV application source. The actual PHP application is downloaded at install time from a GitHub release tarball — currently the **`madtibo/agendav` fork** (see `[resources.sources.main]` and the TODO in `[upstream]` of `manifest.toml`; to be repointed at `agendav/agendav` once it ships 3.0.0). The release asset is self-contained (bundles `web/vendor/` and the compiled `web/public/dist/`), so no composer/npm runs at install time. This repo contains only the manifest, lifecycle scripts, and config templates.

Uses **packaging format v2** (`packaging_format = 2`). The schema is referenced at the top of `manifest.toml` and `tests.toml`.

## Version & branch conventions

- Version format is `<upstream>~ynh<N>` (e.g. `3.0.0~ynh1`). Bump the `~ynhN` suffix in `manifest.toml` for any packaging change; the upstream part tracks the AgenDAV release.
- **PRs target the `testing` branch**, not `master`. `master` is the released branch.
- `README.md` is auto-generated by YunoHost's `readme_generator` — **do not edit by hand**. Edit `doc/DESCRIPTION*.md` and `manifest.toml` instead; the README is regenerated upstream (see the "Auto-update READMEs" commits).

## Architecture & key relationships

**Resources are declarative, not scripted.** In format v2, the system user, install dir, MySQL database, permissions, and APT packages are declared under `[resources]` in `manifest.toml` and provisioned automatically by YunoHost. The lifecycle scripts assume these already exist (variables like `$app`, `$install_dir`, `$db_name`, `$db_user`, `$db_pwd`, `$domain`, `$php_version` are injected by the helpers, not defined in the scripts).

**CalDAV backend discovery is the central piece of custom logic.** AgenDAV is only a frontend — it needs a CalDAV server. `scripts/_common.sh::_ynh_agendav_find_caldav_app()` scans installed YunoHost apps for Baïkal or Radicale and returns `<app_id> <baseurl_suffix>`. `install` and `upgrade` both call this, then build `caldav_baseurl` from the backend app's `domain` + `path` settings. Installation aborts (`ynh_die`) if neither backend is present. `tests.toml` pre-installs Baïkal for CI for this reason.

**Configuration is templated via placeholder substitution.** Since 3.0.0 (Slim 4 + PHP-DI 7), AgenDAV loads config as a chain of PHP files that each `return [ ... ]` of overrides: `web/config/default.settings.php` (defaults) → `web/config/settings.php` (our file) → `web/config/prod.php`. `conf/settings.php` is therefore an **override array** (not the old Pimple `$app['key'] = …` syntax) with `__PLACEHOLDER__` tokens (e.g. `__DB_NAME__`, `__ENCRYPTKEY__`, `__CALDAV_BASEURL__`, `__TIMEZONE__`, `__LANGUAGE__`). `ynh_config_add` substitutes the matching shell variables and writes `$install_dir/web/config/settings.php`. It is regenerated from the template on every upgrade (using the stored `$encryptkey`), so it is intentionally **not** `--kept` across `ynh_setup_source` — the pre-3.0 Silex-format file would be incompatible anyway.

**SSO/HTTP-auth is native, enabled by config.** AgenDAV 3.0.0 ships `\AgenDAV\Authentication\HttpBasic` (in `web/src/Authentication/`), which reads `PHP_AUTH_USER`/`PHP_AUTH_PW`. `conf/settings.php` enables it via `'auth.methods' => [\AgenDAV\Authentication\HttpBasic::class]`, so users authenticate transparently from YunoHost's SSO. `conf/nginx.conf` must forward the credentials: `fastcgi_param REMOTE_USER $remote_user;` **and** `fastcgi_param HTTP_AUTHORIZATION $http_authorization;` (HttpBasic requires the `HTTP_AUTHORIZATION` server param). There is **no source patch** anymore (the old `app-00-add-http-auth.patch` against Silex's `web/app/controllers.php` was removed).

**The session encryption key is managed.** AgenDAV encrypts the CalDAV password it keeps in each session (`web/app/services.php` `password.cipher`). `conf/settings.php` sets `session.encryption.key` from a generated `session_key` app setting (`openssl rand -hex 32`, created in `install`, backfilled in `upgrade`). This is **required**, not optional: the upstream fallback writes a key file to `dirname(log.path)/session.key` = `/var/log/session.key` (our `log.path` is `/var/log/$app/`), which the app user can't create — leaving it unset breaks the first login.

**Database migrations run on both install and upgrade** via `php$php_version agendavcli migrations:migrate --no-interaction` (this also initializes the schema on first install).

**Permissions are security-sensitive.** `web/config/` holds CalDAV passwords and is locked to `750` / `$app:www-data` (group-readable by the web server, not world-readable). `web/var/cache/{profiler,twig}` is owned by `$app` so the app can write it. Backup cleans these cache dirs before archiving.

## YunoHost version & helpers

Targets **YunoHost ≥ 12.1.17** (`[integration]`) and uses **helpers v2.1**. This requires `helpers_version = "2.1"` under `[integration]` in `manifest.toml`: without it YunoHost 12 loads the **2.0** helpers and every `ynh_config_*` call dies with `command not found` (exit 127). The 2.1 set means positional `ynh_script_progression "msg"` / `ynh_die "msg"` (no `--message`/`--weight`), the `ynh_config_add*` / `ynh_config_remove*` namespace, `ynh_systemctl`, `ynh_safe_rm`, `ynh_backup`/`ynh_restore` (positional paths), and `ynh_mysql_dump_db` / `ynh_mysql_db_shell`. Do not reintroduce the YNH 11 names (`ynh_add_config`, `ynh_systemd_action`, `ynh_secure_remove`, `ynh_check_app_version_changed`, etc.) — the upgrade script intentionally has no `upgrade_type` guard and always re-runs `ynh_setup_source`.

## PHP version constraint

AgenDAV 3.0.0 requires **PHP ≥ 8.5** (its only supported runtime). `[resources.apt]` pins `php8.5-*` packages. `$php_version` (note the underscore — renamed from `$phpversion` in helpers 2.1) is derived automatically from those package names, so keep the `php8.5-` prefix consistent across all apt entries if the upstream PHP requirement changes. **Caveat:** YunoHost 12 / Debian 12 defaults to PHP 8.2; php8.5 is pulled from the Sury extra-repo that the apt resource configures — verify `php8.5-fpm` actually installs on the target.

## Testing & development

There is no local build/lint/test toolchain — validation happens through YunoHost itself.

Test the `testing` branch on a YunoHost instance:
```bash
# fresh install (a CalDAV backend like Baïkal must be installed first):
sudo yunohost app install https://github.com/YunoHost-Apps/agendav_ynh/tree/testing

# upgrade an existing install:
sudo yunohost app upgrade agendav -u https://github.com/YunoHost-Apps/agendav_ynh/tree/testing
```

Automated tests run on YunoHost's CI per `tests.toml` (which defines the Baïkal pre-install; there is no upgrade-from test since the 3.0.0 rewrite). After editing the manifest or scripts, lint the package with [`package_check`](https://github.com/YunoHost/package_check) / the YunoHost app linter rather than expecting a script in this repo.

Source updates (`url`/`sha256` in `[resources.sources.main]`) are handled automatically by `autoupdate.strategy = "latest_github_tag"`.
