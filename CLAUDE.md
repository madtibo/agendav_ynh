This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

This is a **YunoHost packaging wrapper** for AgenDAV (a CalDAV web client), **not** the AgenDAV application source. The actual PHP application is downloaded at install time from upstream GitHub releases (see `[resources.sources.main]` in `manifest.toml`). This repo contains only the manifest, install/upgrade/backup lifecycle scripts, config templates, and patches.

Uses **packaging format v2** (`packaging_format = 2`). The schema is referenced at the top of `manifest.toml` and `tests.toml`.

## Version & branch conventions

- Version format is `<upstream>~ynh<N>` (e.g. `2.6.0~ynh5`). Bump the `~ynhN` suffix in `manifest.toml` for any packaging change; the upstream part tracks the AgenDAV release.
- **PRs target the `testing` branch**, not `master`. `master` is the released branch.
- `README.md` is auto-generated by YunoHost's `readme_generator` — **do not edit by hand**. Edit `doc/DESCRIPTION*.md` and `manifest.toml` instead; the README is regenerated upstream (see the "Auto-update READMEs" commits).

## Architecture & key relationships

**Resources are declarative, not scripted.** In format v2, the system user, install dir, MySQL database, permissions, and APT packages are declared under `[resources]` in `manifest.toml` and provisioned automatically by YunoHost. The lifecycle scripts assume these already exist (variables like `$app`, `$install_dir`, `$db_name`, `$db_user`, `$db_pwd`, `$domain`, `$phpversion` are injected by the helpers, not defined in the scripts).

**CalDAV backend discovery is the central piece of custom logic.** AgenDAV is only a frontend — it needs a CalDAV server. `scripts/_common.sh::_ynh_agendav_find_caldav_app()` scans installed YunoHost apps for Baïkal or Radicale and returns `<app_id> <baseurl_suffix>`. `install` and `upgrade` both call this, then build `caldav_baseurl` from the backend app's `domain` + `path` settings. Installation aborts (`ynh_die`) if neither backend is present. `tests.toml` pre-installs Baïkal for CI for this reason.

**Configuration is templated via placeholder substitution.** `conf/settings.php` is the upstream `settings.php` with `__PLACEHOLDER__` tokens (e.g. `__DB_NAME__`, `__ENCRYPTKEY__`, `__CALDAV_BASEURL__`, `__TIMEZONE__`, `__LANGUAGE__`). `ynh_config_add` substitutes the matching shell variables and writes to `$install_dir/web/config/settings.php`. On upgrade, this file is preserved across source re-download via `ynh_setup_source --keep="web/config/settings.php"`.

**SSO/HTTP-auth integration is applied as a source patch.** `sources/patches/app-00-add-http-auth.patch` is applied automatically by `ynh_setup_source` during install/upgrade. It modifies AgenDAV's `web/app/controllers.php` and `web/src/Controller/Authentication.php` so the app authenticates from `$_SERVER['PHP_AUTH_USER']`/`PHP_AUTH_PW` (set by YunoHost's SSO) instead of only its own login form. The nginx config forwards `$remote_user` via `fastcgi_param REMOTE_USER`.

**Database migrations run on both install and upgrade** via `php$php_version agendavcli migrations:migrate --no-interaction` (this also initializes the schema on first install).

**Permissions are security-sensitive.** `web/config/` holds CalDAV passwords and is locked to `750` / `$app:www-data` (group-readable by the web server, not world-readable). `web/var/cache/{profiler,twig}` is owned by `$app` so the app can write it. Backup cleans these cache dirs before archiving.

## YunoHost version & helpers

Targets **YunoHost ≥ 12.0** (`[integration]`) and uses **helpers v2.1**: positional `ynh_script_progression "msg"` / `ynh_die "msg"` (no `--message`/`--weight`), the `ynh_config_add*` / `ynh_config_remove*` namespace, `ynh_systemctl`, `ynh_safe_rm`, `ynh_backup`/`ynh_restore` (positional paths), and `ynh_mysql_dump_db` / `ynh_mysql_db_shell`. Do not reintroduce the YNH 11 names (`ynh_add_config`, `ynh_systemd_action`, `ynh_secure_remove`, `ynh_check_app_version_changed`, etc.) — the upgrade script intentionally has no `upgrade_type` guard and always re-runs `ynh_setup_source`.

## PHP version constraint

The app pins **PHP 7.4** packages (`php7.4-*` in `[resources.apt]`). `$php_version` (note the underscore — renamed from `$phpversion` in helpers 2.1) is derived automatically from those package names, so keep the `php7.4-` prefix consistent across all apt entries if the upstream PHP requirement changes.

## Testing & development

There is no local build/lint/test toolchain — validation happens through YunoHost itself.

Test the `testing` branch on a YunoHost instance:
```bash
# fresh install (a CalDAV backend like Baïkal must be installed first):
sudo yunohost app install https://github.com/YunoHost-Apps/agendav_ynh/tree/testing

# upgrade an existing install:
sudo yunohost app upgrade agendav -u https://github.com/YunoHost-Apps/agendav_ynh/tree/testing
```

Automated tests run on YunoHost's CI per `tests.toml` (which defines the Baïkal pre-install and the upgrade-from commit). After editing the manifest or scripts, lint the package with [`package_check`](https://github.com/YunoHost/package_check) / the YunoHost app linter rather than expecting a script in this repo.

Source updates (`url`/`sha256` in `[resources.sources.main]`) are handled automatically by `autoupdate.strategy = "latest_github_tag"`.
